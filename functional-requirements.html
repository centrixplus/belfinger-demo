How to use this document

This document lists every feature we plan to build in Phase 1. It pairs with the Scope of Work (which covers architecture, timeline, and commercials) and the Clickable Prototype (which shows what each screen will look like).

For each of the 113 features below, please mark one of the four approval states by ticking the box. Write any changes in the margin or on an accompanying note.

Approval legend

Build as specified — no changes required

✗

How approval works: any feature marked "Change" or "Discuss" will be resolved in a working session with the implementation team before Phase 2 (Detailed Planning) begins. All "Agreed" features move directly into the build plan. "Not needed" features are excluded from Phase 1 and do not affect timeline or cost.

Approval Chain & Routing
Enforced state machine from MEW down to Site Engineer
6 features
IDFeatureWho uses✓✎✗?
F6.1Forward approval chain
MEW → Belfinger Mgr → Contractor Mgr → Supervisor → Site Engineer. Enforced by system.
System
F6.2Skip-contractor-mgr rule
If dept owned by Belfinger (e.g. Water), the Contractor Mgr step is skipped automatically.
System
F6.3Visible state on every WO
Always shows current state + current actor (whose court the ball is in). No guesswork.
All users
F6.4Permit enforcement
Cannot transition to ASSIGNED_TO_ENGINEER without Work Permit attached. No bypass.
System
F6.5Supervisor engineer assignment
Supervisor picks engineer from available team. List shows open WO count + site distance.
Supervisor
F6.6Engineer acceptance
Engineer can acknowledge or reject back to Supervisor with reason (wrong assign, safety, no permit).
Site Engineer
7
Rejection & Rework
"Choose who to send it back to" logic with constrained targets
5 features
IDFeatureWho uses✓✎✗?
F7.1Forward-chain rejection
Any approver can reject back upstream with mandatory reason.
Any approver
F7.2"Choose who to send back to"
Approver picks specific target from dropdown of legal upstream roles. Not just one step back.
Approver
F7.3Constrained targets by role
Can only reassign upstream — never sideways. Prevents blame-ping-pong.
System
F7.4Mandatory rejection reason
System refuses empty reason. Stored in audit log and shown in all later views.
System
F7.5Rework returns to IN_PROGRESS
Rejected work re-enters IN_PROGRESS on same WO (not a new one). More data added.
System

Out of Scope (Phase 1)

These items are not included in Phase 1. They can be added in a future phase if required.

O1
MEW finance / GL integration
Invoices exported as PDF for manual handover in Phase 1. Direct API integration can be added in Phase 2.
O2
Subcontractor → Belfinger pricing
Handled offline in v1. Only the MEW-approved BOQ is tracked.
O3
Predictive maintenance & analytics
Needs historical data first. Best for Phase 2 after 6+ months of live data.
O4
Native iOS / Android apps
PWA covers iPad and Android tablet without App Store review.
O5
Real-time DCS / SCADA telemetry
Phase 2 feature once MEW's DCS APIs are understood.
O6
Parts / inventory management
v1 captures "spare installed" as a text field. Full parts module is a future phase.
O7
Financial accounting (GL, AR, AP)
Out of scope — use Belfinger's existing accounting system.
